## Full Text
# ТЗ (черновик): MPC/TSS библиотека для кастоди‑сервиса (EOA, мультичейн)  
## Профиль: **Trusted‑HSM**, СКЗИ — на стороне заказчика (периметр), подписи максимально упрощены

**Версия:** v0.2 (на основе текущей переписки)  
**Статус:** draft для правок

---

## 1. Цель документа
Зафиксировать бизнес‑ и продуктовые требования к внутренней MPC/TSS библиотеке для кастоди‑сервиса, ориентированной на **EOA‑кошельки** в нескольких блокчейнах (Ethereum, Bitcoin, Tron и др.), с упором на эксплуатацию в **доверенной среде заказчика** (HSM/изолированный контур).  
СКЗИ (сертифицированные средства защиты каналов/хранилищ/аудита) находятся **в инфраструктуре заказчика**, не являются частью поставки библиотеки.

---

## 2. Контекст и ключевые допущения

### 2.1 Форм‑фактор и аудитория
- Библиотека — **внутренняя** (internal), используется в составе кастоди/WaaS стека (обычно on‑prem у клиента).
- Основные пользователи — банки/институционалы, где инфраструктура подписи разворачивается в защищённом контуре.

### 2.2 Аккаунт‑модель
- Только **EOA** (без AA / смарт‑контрактных аккаунтов).
- Подписи должны быть **нативно валидны** для сетей (никакой замены на ГОСТ‑подписи).

### 2.3 Профиль безопасности **Trusted‑HSM**
- Доли ключа (shares) и чувствительные материалы (presign, nonce‑материалы) находятся:
  - в HSM/в изолированной доверенной среде (по политике заказчика), **или**
  - в процессах, защищённых периметром заказчика (СКЗИ/сегментация/контроль доступа).
- Мы **упрощаем** протокол подписи, исходя из модели угроз: нет активного malicious‑участника внутри кворума (по умолчанию **semi‑honest / fail‑stop**), но есть риск сбоев/отвалов/повторов сообщений и внешних атак (перехват/подмена) — они должны быть закрыты периметром и протокольной защитой от replay/out‑of‑order.

> Примечание: при необходимости заказчика допускается «Hardened profile» (malicious 1‑party, ZK‑пруфы/строгие проверки), но он **не является P0** для текущего ТЗ.

---

## 3. Скоуп

### 3.1 In‑Scope (P0)
1) MPC/TSS‑ядро для ECDSA(secp256k1) в режиме **2‑of‑3**.  
2) DKG/KeyGen (без trusted dealer): совместная генерация ключевого материала в долях.  
3) Signing: подпись дайджестов и/или транзакций для:
   - Ethereum (EIP‑155, low‑S, `r,s,v`),
   - Bitcoin (DER signature, sighash),
   - Tron (ECDSA/secp256k1).
4) **Presigning (preprocessing)**: обязательный пул одноразовых материалов для ускорения подписи.  
5) **HD Wallet**: поддержка деривации ключей (BIP32/BIP44‑совместимые пути для BTC‑семейства; стандартные пути для ETH/TRON).  
6) **Refresh** и **Reshare**:
   - refresh: обновление долей без смены публичного ключа,
   - reshare: замена участника/устройства/модуля (например, новый HSM/новая нода) без смены публичного ключа.
7) Runtime/Session Engine: управление сессиями, retry/timeout, pause/resume, idempotency.  
8) Audit‑events (без секретов) + hooks для внешнего подписания/штампования логов средствами заказчика.

### 3.2 Out‑of‑Scope (явно)
- Policy/approvals, «permit»‑механика, KYT/санкции, оркестрация бизнес‑процессов (это слой продукта).
- Реализация СКЗИ (сертифицированный VPN/TLS/хранилища/лог‑штампы) — **вне библиотеки**.
- Broadcast транзакций в сеть и управление нодами.

---

## 4. Бизнес‑требования (BR)

**BR‑01. Мультичейн EOA.** Поддержка минимум ETH/BTC/TRON и расширяемость под новые сети.  
**BR‑02. Custody‑grade ключи.** Приватный ключ никогда не существует целиком в одном месте.  
**BR‑03. Производительность.** Подпись должна быть совместима с high‑throughput контуром (за счёт presign).  
**BR‑04. Жизненный цикл.** Обязательны операции refresh/reshare без смены публичного ключа (минимизация “переводов на новый адрес”).  
**BR‑05. Совместимость с СКЗИ‑периметром заказчика.** Библиотека должна интегрироваться с внешним защищённым каналом/хранилищем/RNG/аудитом.  
**BR‑06. Расследуемость.** Достаточный аудит для расследований и комплаенса без утечки секретов.

## 5. Продуктовые требования (PR) — функционал

### 5.1 Поддерживаемые роли/участники (2‑of‑3)
- `PartyA` — основной подписант (HSM/сервис).
- `PartyB` — второй подписант (HSM/сервис).
- `PartyR` — recovery/backup подписант (HSM/изолированный контур).

> Примечание: если в будущем нужен «mobile party», это отдельный профиль (не P0 в Trusted‑HSM).

### 5.2 DKG / KeyGen (P0)
**PR‑DKG‑01.** Реализовать DKG для 2‑of‑3.  
**PR‑DKG‑02.** Результат: публичный ключ + адреса (ETH/BTC/TRON) + share‑артефакты.  
**PR‑DKG‑03.** Минимальные проверки целостности (VSS/commitments или эквивалент) без тяжёлых ZK‑пруфов в Trusted‑HSM профиле.

### 5.3 Signing (P0)
**PR‑SIG‑01.** Подпись дайджеста `msg32` ECDSA/secp256k1 в 2‑of‑3.  
**PR‑SIG‑02.** Адаптеры формата подписи:
- ETH: `(r,s,v)` + low‑S + EIP‑155,
- BTC: DER + sighash,
- TRON: форматы сети.  
**PR‑SIG‑03.** Минимум онлайновых раундов за счёт presign.

### 5.4 Presigning (обязателен, P0)
**PR‑PRE‑01.** Генерация presign‑пакетов заранее, хранение в `PresignStore`.  
**PR‑PRE‑02.** Жизненный цикл: `AVAILABLE → RESERVED → SPENT/BURNED`.  
**PR‑PRE‑03.** Abort/timeout ⇒ `BURN` всех затронутых одноразовых материалов.  
**PR‑PRE‑04.** Идемпотентность: никакого повторного расходования пакета.

### 5.5 HD Wallet (P0)
**PR‑HD‑01.** MPC‑совместимая HD деривация (BTC BIP32/BIP44‑семантика; ETH/TRON пути).  
**PR‑HD‑02.** Derive addresses/xpub (где допустимо) + signing по пути.  
**PR‑HD‑03.** Кэширование без утечки секретов.

### 5.6 Refresh / Reshare (P0)
**PR‑REF‑01.** Refresh без смены публичного ключа.  
**PR‑RSH‑01.** Reshare (замена party/модуля) без смены публичного ключа.  
**PR‑RSH‑02.** После recovery процедур reshare/refresh — обязательны (механизм предоставляется библиотекой).

## 6. Runtime, API и протокол сообщений

### 6.1 Session Engine (P0)
**PR‑SES‑01.** Все операции как сессии с `session_id`.  
**PR‑SES‑02.** Retry/timeout/pause‑resume/dedup.  
**PR‑SES‑03.** Защита от replay/out‑of‑order (nonce/counters + session binding).

### 6.2 API (P0, примерный контракт)
- `keygen_init(...) -> session_id`
- `keygen_step(session_id, msg) -> msgs[] | done(pubkey, addresses)`
- `presign_init(wallet_id, quorum, count) -> session_id`
- `sign_init(wallet_id, path, msg32, quorum) -> session_id`
- `*_step(session_id, msg) -> msgs[] | done(result)`
- `refresh_init(...)`, `reshare_init(...)` аналогично

---

## 7. Требования безопасности (SEC) — Trusted‑HSM

### 7.1 Свойства (P0)
**SEC‑01. Never reconstruct**: полный приватный ключ не реконструируется.  
**SEC‑02. Nonce/presign safety**: single‑use, burn‑on‑abort.  
**SEC‑03. Session binding**: все сообщения привязаны к `session_id` и контексту.  
**SEC‑04. No secret logging**: запрет на секреты в логах/трейсах.  
**SEC‑05. Secure deletion**: best‑effort zeroization и secure delete для presign.  
**SEC‑06. Identifiable abort (must)**: диагностировать «кто/где сорвал» (по шагу и identity).

---

## 8. СКЗИ‑периметр: границы и требования к интеграции

### 8.1 Граница ответственности
- СКЗИ (сертифицированные VPN/TLS, хранилища, подпись логов) реализует **заказчик**.
- Библиотека обязана иметь **адаптеры/хуки**, но не поставляет СКЗИ.

### 8.2 Интеграционные требования (SKZI‑REQ)
**SKZI‑01.** Работать поверх внешнего secure transport, требовать аутентифицированные peer identities.  
**SKZI‑02.** Anti‑replay/out‑of‑order на уровне протокольных сообщений (даже поверх защищённого канала).  
**SKZI‑03.** `KEKProvider`: wrap/unwrap для секретных blob’ов + AEAD at‑rest.  
**SKZI‑04.** `RngProvider`: возможность внешнего RNG.  
**SKZI‑05.** Экспорт audit‑events в формат, пригодный для внешнего подписания/штампования.

## 9. HSM интеграция

### 9.1 Режимы
- **HSM‑MODE‑A (P0)**: HSM хранит shares/KEK (storage‑only), MPC‑операции выполняются в доверенном процессе.  
- **HSM‑MODE‑B (P1)**: compute‑capable (RNG/scalar ops/unwrap внутри модуля).

### 9.2 Требования
**HSM‑01.** Non‑exportability секретов.  
**HSM‑02.** Wrap/unwrap через KEKProvider.  
**HSM‑03.** Учет ограничений HSM на параллелизм (presign/sign).  
**HSM‑04.** Audit по операциям unwrap/sign‑step.

---

## 10. Тестирование и DoD

### 10.1 Тесты (P0)
1) DKG → адреса ETH/BTC/TRON.  
2) HD derivation: несколько путей, проверка адресов.  
3) Presign pool: reserve/spend/burn.  
4) Signing: верификация стандартными верификаторами.  
5) Abort на каждом шаге: burn + identifiable abort.  
6) Refresh: публичный ключ тот же, shares новые.  
7) Reshare: замена party без смены публичного ключа.

### 10.2 DoD (P0)
- Реализованы DKG/sign/presign/HD/refresh/reshare/session engine/audit.  
- Пройдены e2e + негативные тесты.  
- Документация API + state machine + storage/transport contracts.  
- Подтверждено отсутствие секретов в логах.
